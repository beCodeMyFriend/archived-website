Con un desayuno en el Hotel... comienza el Code Retreat. agilismo.es con el patrocinio de Autentia y Eden Development,organiza el evento, con Enrique Comba al frente.

Durante el evento contamos de alguna manera con la presencia de Corey Haines que, mediante una videoconferencia, nos saludó y pudimos intercambiar algunas palabras con el impulsor de los code retretas.

El reto propuesto está basado en el conocido "Juego de la Vida" , de John Horton Conway.

En los Code Retreats, se trabaja en iteraciones de 45 minutos, borrando todo el código anterior antes de cada una y cambiando de pareja cada vez.

Las 2 primeras iteraciones suponen una aproximación al problema. En los intervalos entre una iteración y la siguiente, Enrique empieza a hablar del "naming"  y el resto de los asistentes participamos dando nuestra opinión hasta llegar a conclusiones como la importancia que tiene encontrar el nombre adecuado de cada una de las variables. Es decir, si somos capaces de escribir de manera entendible nuestro código, será mucho más fácil entender el problema, tanto para nosotros como para aquellos programadores que lo lean.

Un ejemplo de código sin un buen naming:

public function test1()
{
$myVar = 0;
$object = new Cell();
$result = $object->getStatus($myVar);
$this->assertFalse($result);
}
no hay forma de saber que hace este test, sin embargo algo así:

public function testCheckIfCellDiesWithZeroNeighbours()
{
$neighbours = 0;
$cell = new Cell();
$cellIsAlive = $cell->checkLife($neighbours);
$this->assertFalse($cellIsAlive);
}
Como veis, el segundo test es el mismo que el primero, sin embargo queda mucho mas claro que es lo que estoy testeando.

La tercera iteración supuso un antes y un después en la manera de entender el TDD. Para empezar, la duración se redujo a 20 minutos. Tiempo en el cual Enrique nos plantea no crear clases ni métodos, sino escribir el código necesario para pasar un test dentro del propio test.

Transcurridos esos 20 minutos se realiza un braimstorming en el cual hay diferentes opiniones entre los participantes, que van  desde el desconcierto hasta emoción ante el nuevo reto. Antes de pasar a la siguiente iteración Enrique formula dos "code restrains".

- Escribir nuestros tests, incluido el código para resolverlo no debe superar las cuatro líneas.

- No se pueden utilizar "if"s.

Nuestros test quedaron algo así:

public function testCellWithLessThan2NeighboursDies()
{
$neighbours = 0;
$minimumNeightboursForKeepAlive = 2;
$cellIsAlive = $neighbours >= $minimumNeighboursForKeepAlive;
$this->assertFalse($cellIsAlive);
}

Tras esta iteración que yo llamaría "Extreme TDD" concluimos que, aunque la mayoría de veces creemos estar haciendo buen TDD, en realidad ya estamos cerrados a una solución puesto que testeamos los objetos que creemos que vamos a necesitar, como en las iteraciones anteriores, que comenzamos creando objetos como "tablero", "célula" o "universo". En cambio, de esta manera nos mantenemos más en el espacio del problema, lo que, después de refactorizar, nos lleva a soluciones más simples y eficaces.

A lo largo del día pude "pairear" con expertos en otros lenguajes de programación como por ejemplo Ruby; agradezco mucho a Luismi que me enseñase. No faltó el humor a la hora de plantear las soluciones desde un punto de vista casi "divino" cuando alguien creo el objeto "Dios" u otros que preferian utilizar a "La Madre Naturaleza" para crear vida!.
Fue un día muy entretenido para mi, una experiencia que espero poder repetir.
